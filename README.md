# Поиск статей справочного центра Авито

Выполнила: Гуреева Алина

Задание - по вопросу пользователя найти подходящие статьи из справочного центра Авито и вернуть топ-10 article_id. Качество оценивается метрикой MAP@10.

---

## Описание решения

### Как я обрабатывала HTML статей

Тела статей приходят в сыром HTML. Просто вызвать `.get_text()` не получилось, пункты списков слипались в одну строку без пробелов, например `<li>Шаг 1</li><li>Шаг 2</li>` превращалось в `"Шаг 1Шаг 2"`.

Я решила это так: перед удалением тегов прохожусь по всем `<li>`, `<p>`, `<div>`, `<br>`, `<tr>` и вставляю в конец каждого временный разделитель `" | "`. После извлечения текста заменяю его на точку. Итог: `"Шаг 1. Шаг 2."`, читаемо и не слипается.

Дополнительно убираю:
- ссылки (они шумят и не несут смысла для поиска)
- эмодзи и типографские иконки типа ⚡★ (unicode-мусор)
- двойные точки и пробелы перед знаками препинания, которые появлялись после всех замен
- `&nbsp;` и другие HTML-сущности через `html.unescape`

Заголовок статьи я дублирую дважды перед телом, так при поиске он получает больший вес.

### Какие модели и алгоритмы использовала

**Dense embeddings**: `intfloat/multilingual-e5-large`. Нейросеть переводит текст в числовой вектор из 1024 чисел так, что близкие по смыслу тексты оказываются рядом в пространстве. Например, "удалить объявление" и "снять публикацию" найдут одну и ту же статью, даже если слова не совпадают. Модель мультиязычная, хорошо работает на русском. При индексации использую префикс `"passage: "`, при поиске `"query: "`, этого требует архитектура e5.

**Sparse embeddings (BM25)**: через `Qdrant/bm25` из fastembed. Классический алгоритм поиска по ключевым словам: чем реже слово встречается в базе и чем чаще в конкретном документе, тем выше его вес. Хорошо ловит точные термины, числа, названия, то, с чем dense иногда промахивается.

**RRF (Reciprocal Rank Fusion)**: алгоритм объединения двух списков результатов. Каждая статья получает очки по формуле `1 / (60 + позиция)` из обоих поисков, потом суммируются. Константа 60 сглаживает разницу между соседними позициями. Это лучше чем просто взять результаты одного поиска.

**Векторная база данных Qdrant**: хранит dense и sparse векторы для каждого чанка, умеет быстро искать ближайших соседей. Запускаю в Docker.

**Chunking**: каждая статья режется на куски по 250 слов с перекрытием 50 слов, потому что модели плохо работают с длинными текстами (ограничение ~512 токенов). Перекрытие нужно чтобы смысл не терялся на границах.

### Как проверяла качество на calibration.f

В `calibration.f` 500 запросов с правильными ответами (`ground_truth`, article_id которые должны быть в топе). Я прогоняю каждый запрос через поиск и считаю AP@10:

- если правильная статья на 1 месте, AP = 1.0
- на 2 месте, AP = 0.5
- на 5 месте, AP = 0.2
- не попала в топ-10, AP = 0.0

MAP@10 это среднее AP@10 по всем 500 запросам. Каждый раз когда меняла модель или параметры, перезапускала и смотрела как изменилась метрика.

### Какие ошибки нашла и что с ними сделала

**Слипание текста из HTML.** Пункты списков объединялись без разделителей. Решила через вставку временного `" | "` перед удалением тегов.

**Приветствия в запросах мешали поиску.** Почти все запросы начинались с "здравствуйте", "подскажите пожалуйста" и т.д. Эти слова уводили поиск не туда, модель искала статьи где есть слово "здравствуйте". Добавила регулярку которая вырезает приветствия перед поиском.

**Reranker на английском ухудшил результат.** Я попробовала reranker `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1`, он обучен на английском датасете MS MARCO. MAP@10 упал с 0.4202 до 0.3219. Для русских запросов нужна мультиязычная модель. Переключилась на `BAAI/bge-reranker-v2-m3`.

**bge-m3 дала результат хуже e5-large.** Несмотря на то что bge-m3 теоретически мощнее (2.3 GB против 560 MB), на нашем датасете e5-large показала 0.4378 против 0.4202 у bge-m3. Возможно из-за особенностей датасета или размера чанков.

**Проблемы со скачиванием больших моделей.** bge-m3 весит 2.3 GB и при скачивании несколько раз обрывалась с CAS Client Error. Добавила Docker volume для кэша HuggingFace, теперь модели скачиваются один раз и сохраняются между перезапусками.

## Структура проекта

```
solution.py       , точка входа: загружает данные, модели, считает MAP@10, пишет answer.csv
cleaner.py        , очистка HTML и запросов
indexer.py        , чанкование и загрузка эмбеддингов в Qdrant
searcher.py       , поиск: dense + sparse + RRF + reranker
evaluate.py       , AP@10 и MAP@10
docker-compose.yml, два контейнера: qdrant и solution
Dockerfile        , образ на python:3.11-slim
requirements.txt  , зависимости
candidate_data/   , данные (articles.f, calibration.f, test.f)
```

---

## Как запустить

```bash
docker-compose up --build
```

При первом запуске скачаются модели (~1.2 GB) и построится индекс. На CPU без GPU это занимает около 2 часов. При повторных запусках всё берётся из кэша.

Если нужно пересобрать индекс (например при смене модели):
```bash
docker-compose down -v
docker-compose up --build
```
