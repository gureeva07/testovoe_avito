# Поиск статей справочного центра Авито

Это тестовое задание, нужно было реализовать поисковый компонент для RAG-пайплайна. Конкретно: по тексту вопроса пользователя найти релевантные статьи из справочного центра Авито и вернуть топ-10 article_id. Качество меряется метрикой MAP@10.

---

## Что я сделала

Я реализовала гибридный поиск: сначала ищу кандидатов через два независимых поиска (смысловой и по ключевым словам), объединяю результаты, а потом переранжирую через reranker.

Вот как выглядит весь пайплайн:

```
Статьи (HTML) → очистка → чанки → индекс в Qdrant
                                        ↓
Запрос → очистка → dense + sparse поиск → RRF → reranker → топ-10
```

### Как устроен поиск

**Dense-поиск** - ищет по смыслу. Я использую модель `intfloat/multilingual-e5-large`, которая переводит текст в числовой вектор. Похожие по смыслу тексты оказываются близко в этом векторном пространстве. Например, "удалить объявление" и "снять публикацию" найдут одну статью, даже если слова не совпадают.

**Sparse-поиск** - ищет по ключевым словам через BM25. Это классический алгоритм: чем реже слово в базе и чем чаще в конкретном документе, тем важнее. Он хорошо ловит точные термины и числа, которые dense иногда упускает.

**RRF (Reciprocal Rank Fusion)** - объединяет оба списка по формуле `1/(60 + позиция)`. Статья получает очки из обоих поисков, итоговый список сортируется по сумме.

**Reranker** `BAAI/bge-reranker-v2-m3` - смотрит на каждую пару (запрос, статья) вместе и выдаёт скор релевантности. Он точнее embedding-моделей, потому что видит связь между словами запроса и текста одновременно. Применяю только к 30 финальным кандидатам, потому что медленный.

### Что я делала с текстами

Статьи приходят в HTML. Я парсю их через BeautifulSoup, вставляю разделители перед удалением тегов чтобы пункты списков не слипались, убираю ссылки и мусорные символы.

Запросы пользователей часто начинаются с "здравствуйте", "подскажите пожалуйста", это не несёт смысловой нагрузки для поиска, поэтому я их вырезаю регулярками.

Заголовок каждой статьи я дублирую дважды перед телом, так он получает больше веса при поиске.

### Почему чанки

Embedding-модели плохо работают с длинными текстами, у них ограничение около 512 токенов. Поэтому я режу каждую статью на куски по 250 слов с перекрытием 50 слов. Перекрытие нужно чтобы не потерять смысл на границах чанков. В индексе каждый чанк хранится отдельно, но с article_id, и при поиске чанки из одной статьи объединяются.

### Почему Docker

Qdrant (векторная БД) запускается как отдельный сервис. Я подняла его через docker-compose вместе со скриптом, так оба контейнера стартуют одной командой, а индекс сохраняется в volume и не пересчитывается при перезапуске.

## Структура проекта

```
solution.py        - точка входа: загружает данные, модели, считает MAP@10, пишет answer.csv
cleaner.py         - очистка HTML и запросов
indexer.py         - чанкование и загрузка эмбеддингов в Qdrant
searcher.py        - поиск: dense + sparse + RRF + reranker
evaluate.py        - AP@10 и MAP@10
docker-compose.yml - два контейнера: qdrant и solution
Dockerfile         - образ на python:3.11-slim
requirements.txt   - зависимости
candidate_data/    - данные (articles.f, calibration.f, test.f)
```

---

## Как запустить

```bash
docker-compose up --build
```

При первом запуске скачаются модели (~1.2 GB) и построится индекс. На CPU без GPU это занимает около 2 часов. При повторных запусках всё берётся из кэша.

Если нужно пересобрать индекс (например при смене модели):
```bash
docker-compose down -v
docker-compose up --build
```

## Что я пробовала и что получилось

| Embedding | Reranker | MAP@10 |
|-----------|----------|--------|
| MiniLM (маленькая) | нет | 0.3250 |
| multilingual-e5-large | нет | 0.4378 |
| BAAI/bge-m3 | нет | 0.4202 |
| BAAI/bge-m3 | mmarco (английский) | 0.3219 |
| multilingual-e5-large | bge-reranker-v2-m3 | в процессе |

Несколько вещей которые я поняла по ходу:
- bge-m3 теоретически мощнее e5-large, но на нашем датасете дала результат хуже. Плюс она весит 2.3 GB и при скачивании несколько раз обрывалась
- Reranker обученный только на английском (mmarco) сделал результат хуже, MAP упал с 0.4202 до 0.3219. Для русского нужна мультиязычная модель
- Без GPU всё считается медленно, 500 запросов с reranker занимают около 1.5-2 часов
